# 0009 — v1.0.1 deferred gates

- **Status:** Proposed (owner decides: tag `v1.0.0` now with documented gaps, or wait)
- **Date:** 2026-05-31

## Context

The pre-v1.0 adversarial sweep returned **GO** (0 Critical, 0 confirmed High) but was scoped to
**static + unit-test** evidence; several Definition-of-Done gates are CI/runtime-gated.
This session (interactive, Docker up, seeded local Supabase) ran the runtime gates that
were previously skipped and self-healed what it found. This ADR records the resulting
scorecard so the `v1.0.0` tag can be cut with **documented** gaps rather than implicit
ones.

## Decision

Tag `v1.0.0` on the gates that are **met with live evidence** (below), and explicitly
**defer the remainder to v1.0.1** as tracked, non-blocking items. None of the deferred
items is a Critical/High security or correctness defect.

### Deferred to v1.0.1 (documented gaps — not blocking the tag)

1. **Lighthouse a11y on *authenticated* pages.** Lighthouse CLI cannot seed the
   client-side Supabase session (`--extra-headers` adds request headers but not the
   `document.cookie` the `AuthGuard` reads), so a CLI run against the room URL redirects
   to `/auth`. **Mitigation: covered by the authenticated axe scan (0 serious/critical),
   which is a stricter WCAG check than Lighthouse's a11y subset.** Closing properly needs
   a Lighthouse **user-flow (puppeteer) script** that logs in first — a v1.0.1 task.
2. **Per-theme authenticated a11y.** The authed axe scan ran against the default theme.
   The other themes (solarized, dark variants) are not yet scanned while authenticated.
   Low risk (the contrast fixes use darker shades / fixed grays), but unverified.
3. **Phase 4 manual sign-offs.** Before/after room-UI screenshots for the PR and a full
   keyboard-only walkthrough of the in-room journey are not captured (automated axe +
   the e2e arrow-key/tablist tests partially cover keyboard).
4. **Sweep re-run on the *merged* `main`.** The GO was on the integration **proxy** tip,
   not a byte-identical merged `main` (sweep caveat #1). Re-run the full panel on merged
   `main` before/after tagging — a process gate, not a known code gap.
5. **`next@14 → 15` upgrade (decision D3).** Six `pnpm audit` advisories; breaking
   upgrade deferred to its own PR + ADR. The CI `audit` job stays **informational
   (allowed-red)** until then.
6. **README badges (Phase 8 DoD).** CI + MIT badges, plus a release badge once tagged —
   trivial, add after the stack lands on `main`.

## Consequences

- **Positive.** Every runtime gate the sweep skipped now has a live PASS except the
  authenticated-Lighthouse number, which is substituted by a stricter axe result. The
  tag can be cut with a precise, honest gap list.
- **Negative / accepted.** Authenticated Lighthouse and the manual UX sign-offs are not
  numerically recorded for v1.0.0. Accepted because authenticated **axe** is 0
  serious/critical and `/auth` Lighthouse is 99.
- **Follow-up (v1.0.1).** Lighthouse user-flow script for authed pages; per-theme authed
  axe; screenshots + keyboard walkthrough; `next@15`; README badges; sweep-on-merged-main.

## Alternatives considered

- **Block the tag until every gate is numerically green (incl. authed Lighthouse + manual
  sign-offs).** Rejected: authed a11y is already covered by axe at a higher bar; gating a
  GO release on a tooling limitation (Lighthouse session seeding) and manual screenshots
  is disproportionate. Better to tag with documented gaps and close them in v1.0.1.
- **Drop the Lighthouse gate entirely.** Rejected: keep it — `/auth` is measured (99) and
  the authed user-flow run is a small, well-scoped v1.0.1 task.
